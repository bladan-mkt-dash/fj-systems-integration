Integration map · required product layer

Operations Reporting Layer

Makes the Admin Check List (daily action report + record ledger) a guaranteed deliverable of systems integration — not an assumed byproduct of the hub, golden record, or appointment sync alone. This is the operational-reporting half of the core project (alongside easier single-entry sync).

Contract: Integration is “done” only when BigQuery holds the tables below and a scheduled Operations Reporting job publishes the Admin Check List views (demographics gaps, upcoming appointment issues, follow-ups, copays, team totals) with date-range filter semantics and remediation state.

Required · demographics completeness contract

Field rules — Missing vs Conflict

Codify which fields matter, which system is authoritative, and how gaps are classified. Without this matrix, sync can succeed while the Admin Check List still has nothing to report.

Field Authoritative Also checked Classification
Date of birth AthenaNet (golden) Zenoti · GHL Missing if blank in Athena; Conflict if peers disagree
Mobile phone AthenaNet Zenoti · GHL Missing / Conflict
Email AthenaNet GHL · Woo (if in scope) Missing / Conflict
Address / ZIP AthenaNet Zenoti Missing / Conflict
Credit card on file Zenoti (primary) GHL flags if used Missing if neither operable card on file

Output table: demographics_gap_fact — patient_id, field, status (missing|conflict), systems_checked, priority, detected_at.

Required · additional data products

Beyond identity & appointments

Hub sync today does not guarantee these. They must be named integration deliverables.

Visit history → follow-ups

  • Last completed visit (date, provider) from AthenaNet (± Zenoti)
  • Recommended follow-up interval rule (care path / provider default)
  • Output: patients overdue or due within N days without a booked follow-up

Balances → outstanding copays

  • Ingest patient balance / open copay from AthenaNet billing (or agreed source)
  • Amount due + aging; attribute to last / responsible provider when possible
  • Output: copay_open_fact

Required · appointment-issue pipeline

Join: upcoming visit + demographic gap + provider

Triggered on appointment ingest/update (and nightly recompute). This is the core of “Upcoming Appointment Issues” on the Admin Check List.

1. Scope visits

Upcoming appointments in the selected date window (default: next 7 days).

2. Evaluate patient

Apply completeness contract against golden record + source fields.

3. Emit issue rows

patient · provider · appt_at · issue_tag · priority (proximity-weighted).

Output table: appt_issue_fact — powers provider lists and the practice-wide summary after date-range filter.

Required · daily snapshot & filter semantics

Scheduled Operations Reporting job

Cadence

Nightly (or every N hours) rebuild of snapshot tables for the Daily Action Report.

Date-range filter

Primary scopes appt issues + practice gap counts tied to upcoming visits. Follow-ups / copays: in-range or unscoped (product choice).

Freshness SLA

Document max lag from source webhook → gap visible on Admin Check List.

Required · remediation state

Assigned · resolved · don’t resurface

Distinct from IAP “who viewed the report.” Front desk / admin must clear gaps without the next daily run repeating finished work.

Assign

Owner (role or person) on gap or issue row.

Resolve

Mark cleared after source system update verified, or waive with reason.

Audit

History of detection → assignment → resolution; suppress open issues that are resolved.

Output table: gap_remediation_state
