Build checklist

Sequential readiness and delivery path for systems integration, ending in the Admin Check List (Daily Action Report) for the 5 Journeys Practice.

5 Journeys 5 Journeys & Contractor Contractor

How to read this: Items are ordered by dependency. Sub-items must be true before the parent is done. “Depends on” points at prior checklist numbers. Architecture context: Operations reporting layer, Admin Check List wireframe. Imagined schedule: Build Gantt (Day −60 → Day 0 launch). Credentials appear only where sequence requires them — store and rotate outside this pack.

1. 5 Journeys

Onus on Five Journeys — decisions, rules, samples, and priorities only you can supply.

  1. Confirm product outcomes and phase priorities
    • Confirm Goals order: unify records → cancels/terminations → Zenoti declines → Admin Check List dashboard as the ops product outcome.
    • Confirm Admin Check List as the near-term live, internal-only operations dashboard (Day 0 go-live gate).
    • Confirm near-term appointment direction: AthenaNet (clinical) / Zenoti (wellness) → GHL + BigQuery only; two-way GHL→AthenaNet deferred.

    Done when a one-page priority note exists that the contractor can quote in the backlog (no open “which first?” debates).

  2. Settle architecture decisions (decision log) Depends on 1
    • AthenaNet = golden source of truth for patient clinical demographics.
    • Woo: identity fields in scope for Goal 1; inventory stays Zenoti↔Woo outside GCP; Woo commerce metrics optional.
    • GHL staff calendars: de-identified busy blocks unless/until full PHI on calendar is approved.
    • Future two-way booking: explicitly gated on Google Meet / telehealth HIPAA–BAA posture (document “not now”).

    Done when each decision above has an owner, date, and status (accepted / deferred) in a short decision log.

  3. Define demographics completeness rules (business) Depends on 2
    • List required fields for Admin Check List (e.g. DOB, phone, email, address/ZIP, credit card on file).
    • For each field: authoritative system + other systems to check.
    • Define Missing vs Conflict in plain language (examples from real messy records welcome).
    • Define priority rules (e.g. High if visit within 7 days).

    Done when a field matrix exists that matches the intent of the operations reporting layer and can be typed into code without reinterpretation.

  4. Define follow-up and copay business rules Depends on 1
    • What counts as “last visit” and which system owns it.
    • How recommended follow-up interval is chosen (care path, provider default, or manual).
    • Which balance/copay source is authoritative for “Outstanding Copays.”
    • Whether follow-ups/copays respect the appointment date-range filter or stay unscoped.

    Done when contractor can write acceptance tests for follow-up and copay rows on the Admin Check List without guessing.

  5. Provide Staff Registry starter list Depends on 2
    • Providers / staff who need GHL calendar reflection of AthenaNet (or Zenoti) appointments.
    • Known IDs where available (AthenaNet provider, Zenoti employee, GHL user) and sync_calendar yes/no.

    Done when Track A can be piloted for at least one clinical provider without inventing roster data.

  6. Supply redacted real-world examples Depends on 3–4
    • 2–3 anonymized “messy patient” stories (missing card, conflicting phone, etc.).
    • 1–2 sample Admin Check List situations you expect to see on a Monday morning.
    • Optional: redacted webhook/API payload screenshots or exports (no secrets).

    Done when examples are attached to the field matrix and dashboard wireframe as reference cases.

  7. Name open questions you are still deciding
    • Woo identity in v1 vs later.
    • Cancel/termination and Zenoti decline timing relative to Admin Check List v1.
    • Any fields that must never appear on a GHL calendar or contact note.

    Done when open questions are listed with “decide by” dates so they don’t silently block build.

2. 5 Journeys & Contractor

Joint workshops and reviews — both sides must participate for the artifact to be accepted.

  1. Walk architecture pack and confirm shared understanding Depends on 1–2; maps on index
    • Review hub, GCP schematic, golden record, appointment sync, operations reporting layer, Admin Check List wireframe.
    • Mark gaps or disagreements on the maps themselves.

    Done when both parties agree the maps describe the intended system (with a dated note of any deferred map updates).

  2. Draft & accept field and event contracts Depends on 3–4, 6, 8
    • Map business field names → system field / API attributes (AthenaNet, Zenoti, GHL, Woo as in scope).
    • List webhook / poll events for patient upsert, appointment booked/updated/cancelled/rescheduled.
    • Agree redacted sample payloads for each critical event.

    Done when a contract doc is signed off (“v1 accepted”) and is the only field reference for engineering.

  3. Draft & accept entity / ID map Depends on 5, 9
    • Golden record keys: athena_patient_id, zenoti_guest_id, ghl_contact_id, optional Woo customer id.
    • Match policy (name + DOB + email) and what happens on ambiguous match.
    • Staff Registry schema and ownership of ongoing updates.

    Done when identity resolution rules are testable and ambiguous-match handling is agreed (manual queue vs block).

  4. Agree Admin Check List acceptance criteria Depends on 3–4, 8
    • Practice summary counts + gap mix behave as specified for a fixed date range.
    • Provider lists: Follow-ups, Outstanding Copays, Upcoming Appointment Issues.
    • Appointment Date Range filter semantics (what is in/out of scope).
    • Remediation: assign / resolve / don’t resurface cleared items.
    • Product posture: live, internal-only staff app (IAP + RBAC) — not patient- or public-facing. HIPAA-conscious GCP posture (BAA’d boundary, BigQuery-only reads).
    • Broader program hygiene in scope for build: least-privilege roles, session timeout, no PHI on GHL staff calendars (de-identified busy blocks), access logging for who viewed which patient report. Excludes physical shoulder-surfing / screen left open in a public space (workforce safeguard).

    Done when criteria are written as checkable statements tied to the Admin Check List wireframe (this is the product DoD for the dashboard). Day 0 means this live internal app meets the bar — not that the static HTML wireframe is “public.”

  5. Agree non-functional envelope Depends on 8
    • Freshness SLA (e.g. appointment → BQ / calendar within N minutes; daily report by time local).
    • Expected volumes; PHI boundaries; idempotency / retry expectations.
    • Environments: sandbox vs production promotion rules.
    • Security hygiene defaults: least-privilege IAM/RBAC, idle session timeout, IAP (or equivalent) for staff-only access, audit logs for report access.
    • Calendar PHI rule: GHL staff calendar events stay de-identified busy blocks unless/until full clinical detail on calendar is explicitly approved.

    Done when SLA, PHI boundary, and hygiene notes are accepted and referenced in the backlog.

  6. Provision access & secrets (joint, when ready to build) Depends on 9–12; only when Phase A is about to start
    • 5 Journeys: create contractor accounts / API apps in AthenaNet, Zenoti, GHL (± Woo) with least privilege.
    • 5 Journeys: issue sandbox credentials; document rotation owner.
    • Contractor: store secrets in agreed vault / secret manager — never in git or HTML docs.
    • Joint: confirm sandbox connectivity with a non-PHI or approved test probe.

    Done when contractor can authenticate to each in-scope sandbox API and 5 Journeys retains ownership of credential lifecycle.

  7. Accept phased delivery plan Depends on 11–12
    • Review Contractor phases A–F below; adjust scope for v1 if needed (e.g. defer declines).
    • Define go-live gate: Admin Check List usable for daily ops on agreed date range.

    Done when phase list and go-live gate are approved by 5 Journeys.

3. Contractor

Design and build — after 5 Journeys inputs and joint acceptances unlock each phase.

Design artifacts (before / alongside early build)

  1. Produce sequence diagrams for critical paths Depends on 9–10
    • Golden-record upsert (happy path + ambiguous match).
    • AthenaNet appointment → BigQuery + GHL staff calendar + patient flags.
    • Daily Admin Check List snapshot job.

    Done when sequences are reviewed in a joint session (item 8 follow-up) and match accepted contracts.

  2. Produce BigQuery data model sketch Depends on 9–11
    • Tables/views: golden record, appointment fact, demographics_gap_fact, appt_issue_fact, copay_open_fact, follow-up fact, gap_remediation_state.
    • Primary keys, grain, and partition/cluster intent.

    Done when model supports every Admin Check List acceptance criterion in item 11 without new mystery tables.

  3. Publish engineering backlog from phases A–F Depends on 14–16
    • Tickets with blockers named (checklist numbers).
    • Test plan stubs per phase (unit / integration / UAT).

    Done when sprint-ready backlog exists and Phase A can start after item 13.

Build phases (sequential)

  1. Phase A — Platform ingest & warehouse Depends on 13, 16–17
    • Stand up middleware + BigQuery project per GCP schematic (HIPAA posture as agreed).
    • Ingest AthenaNet, Zenoti, GHL (± Woo identity) into raw/normalized landing zones.
    • Reporting app can run a trivial authenticated query against BQ.

    Done when sandbox end-to-end “source → BQ → app query” works for at least one entity type per in-scope system.

  2. Phase B — Golden record Depends on 18, 10
    • Identity match + master record in BQ.
    • Fill-if-empty outbound to GHL / Zenoti / AthenaNet (AthenaNet never overwritten when populated).
    • Ambiguous-match handling per item 10.

    Done when pilot patients resolve to one golden ID and blank fields fill per rules; conflicts are detectable for later reporting.

  3. Phase C — Appointments (BQ + one-way calendar / flags) Depends on 19, 5
    • Persist appointment facts (booked / updated / cancelled / rescheduled) in BQ — always.
    • Track A: AthenaNet (and Zenoti as in scope) → staff GHL calendar for linked providers.
    • Track B: patient record flags in GHL / Zenoti.

    Done when a test AthenaNet appointment appears in BQ and on the pilot provider’s GHL calendar (busy block), and cancels update status in BQ.

  4. Phase D — Operations reporting / Admin Check List v1 Depends on 20, 11, 3–4
    • Implement demographics gap detection (Missing / Conflict) from completeness contract.
    • Appointment-issue pipeline joined to upcoming visits + provider.
    • Follow-up and copay facts per business rules.
    • Daily (or agreed cadence) snapshot; Appointment Date Range filter.
    • Remediation state (assign / resolve / suppress).
    • Ship UI or export matching the Admin Check List wireframe.

    Done when item 11 acceptance criteria pass in UAT: authorized staff can use the live internal Admin Check List for a chosen date range without manual spreadsheets — IAP/RBAC enforced, hygiene items from #11–12 verified. This is the primary dashboard go-live gate (Day 0). The static wireframe is design reference only, not the launch surface.

  5. Phase E — Cancellations, terminations & Zenoti declines Depends on 20; may start after or overlapping late D if prioritized
    • Propagate cancel/termination across connected systems per Goal 2.
    • Propagate Zenoti declines per Goal 3.
    • Ensure BQ appointment / membership state stays consistent with Admin Check List.

    Done when agreed cancel/decline scenarios update all in-scope systems and no longer appear as active upcoming issues incorrectly.

  6. Phase F — Future two-way booking (explicitly deferred) Depends on 2 (Meet/HIPAA decision) + stable Phases C–D
    • Design reverse path: GHL patient booking → AthenaNet appointment.
    • Do not build until 5 Journeys accepts Google Meet / telehealth HIPAA posture.

    Done when either (a) documented deferral remains in force, or (b) after compliance go-ahead, reverse sync passes a joint UAT without breaking AthenaNet-as-source-of-record rules.

End goal

Integration is successful when the Admin Check List is a live, internal-only, HIPAA-conscious ops dashboard for the 5 Journeys Practice — staff access only (IAP + RBAC), BigQuery-backed, filterable by appointment date range — with accurate demographics gaps, provider action lists, and remediation, as guaranteed by the operations reporting layer. Not a patient- or public-facing page. Day 0 is when that live app meets the acceptance bar, not when the static HTML wireframe is shared. Hub, golden record, and appointment sync are means to that end; Phase D is the dashboard finish line for v1. Program hygiene (least privilege, session timeout, no PHI on GHL staff calendars, access logs) is in scope; physical shoulder-surfing remains a workforce control outside the app.
